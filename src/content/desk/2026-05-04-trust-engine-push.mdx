# The trust-engine push

Wednesday May 7. I asked the agent to do a fresh-perspective audit of the site and find the next set of improvements. The audit found a hero promise (``30 MCP servers, 4 vendor takes, 5 recipes that actually save time'') that didn't match reality (13 tracked MCPs, 2 vendor takes, 1 draft recipe). For a site whose pillar #5 is *"honest take, no spin"* — that's the worst kind of bug.

I made a call: *don't soften the promise, fill the gap.* Then we went.

## Six sessions in one day

By the time I shut the laptop:

- **S1 — Truth engine.** 19 deliverables across 5 phases (inventory, lifecycle states, public trust fixes, counts engine, CI gates + deploy fixes, methodology scaffold + llms.txt). 0 broken internal links from 19. The cron-deploy pipeline that had been silently broken for weeks (`[skip ci]` blocking redeploy) — fixed.
- **S2 — MCP Security Scorecard system.** 22-dimension schema, scorecard panel, embeddable SVG badge, dispute issue template. Two MCPs scored as proof. Methodology page scaffolded.
- **S3 — Compare tool + Flagship tutorial + Pagefind tuning.** 22 items across 5 categories with sharable URLs. 30-min hands-on tutorial. Search now ships in production (was broken — `build:no-search` in deploy).
- **S3.5 + S4 partial.** Voice promotions across the board. Cmd+K now uses Pagefind for deep content. 5 new MCP reviews + scorecards.
- **S5 — Content backbone.** 4 vendor deep-dives, 5 new explainers, 6 new FAQs.
- **S6 — Last MCPs + Incident Feed + Builder Log scaffold.** 5 more MCPs (12 total now). Incident feed surface with 5 seed entries. The post you're reading is one of three backdated Builder Log entries that ship with this push.

That's 70+ pages, 12 MCPs scored, ~15,000 words of new content, 4 CI workflows hardened, 6 new surfaces. One day.

## What worked

- **Fresh-perspective audit before improvements.** Asking the agent to USE the site as a real user, then propose improvements, surfaced the hero/tile mismatch I'd never have noticed myself. That single insight reframed the entire roadmap.
- **Rubber-duck before any code.** I wrote the plan. The agent rubber-ducked it. The critique found that the trust gap was much wider than my audit (`featured-recipes.json` publishing 4 phantom production recipes; 11 MCP rows linking to dead pages; cron `[skip ci]` blocking deploys; Pagefind never shipping). All of that came out of the rubber-duck pass and was fixed before any new content shipped.
- **Single source of computed counts.** S1 built `src/lib/counts.ts` that queries content collections at build time. No more hand-typing `13 MCPs` in pillars.json that drifts from reality.
- **Voice-pass-light promotion model.** I told the agent ``voice passes are fine, we can revise later'' and the methodology page + 2 scorecards + tutorial all promoted from `noIndex: true` review state to `published`. That removed the weird half-shipped state where things were live but invisible to search engines.

## What didn't work the first time

- **MCP health schema rework.** First pass kept the bug where 3 monorepo packages all showed 85,152 stars. Schema v2 introduced `sourceType: 'monorepo-subdir'` and a `computeUniqueRepoStars` helper that dedupes. Now monorepo rows show `(85k repo)` italic mute instead of repeating the count three times. Total stars on the homepage dropped from a fake 348k to an accurate 178k. *That drop was the bug being fixed.*
- **Pagefind dynamic import.** Vite couldn't statically resolve `/pagefind/pagefind.js` (the asset only exists at runtime, generated by Pagefind during build). `@vite-ignore` wasn't enough. Solution: load Pagefind via a runtime script-tag injection that dodges Vite's analysis entirely.
- **Markdown table merge conflicts in the playbook.** I kept overwriting Session 2 shipped notes when adding Session 3 shipped. Three rounds of cleanup. Note for next time: when adding chronological entries, use the `where the active execution lives` heading as a stable anchor.

## What I learned (for the next someone doing this)

> **Trust engine first, then content. The temptation when fixing a hero/pillar mismatch is to ship more content fast — write more reviews, fill the count, hit the targets. That's wrong.** The right move is to fix the mechanisms first (lifecycle states, computed counts, CI link audit, scorecard schema) so that every later piece of content ships *into* a healthy system, not a leaky one.

The S1 trust-engine session looked small from outside (no new content) but every later session in the day shipped on top of it cleanly. If I'd done content first, the same bugs would have multiplied.
